How it works

Not an autonomous black box. A live shell workflow with a model in the loop.

01

Read local context

Shellia starts from your actual directory, shell, git state, and recent session memory.

02

Plan explicitly

The model proposes concrete commands, Shellia reclassifies them locally, and you can inspect or edit them before they run.

03

Observe and re-plan

If later steps depend on real output, Shellia can pause, inspect, and continue with a better-informed next round.

04

Stay in the same session

Prompt mode, ! commands, and :shell all keep sharing the same working context.

Runtime

One tool, two manual engines, one consistent session.

Shellia supports prompt mode for AI planning and direct command execution when you already know what you want to run.

  • you › for natural-language requests.
  • !<cmd> for one direct command.
  • :shell for a manual shell loop inside the same context.
  • :ai to return to prompt mode.
  • y / e / i / n to run, edit, force interactive, or cancel AI-proposed steps.
[execution]
shell_mode   = "interactive"
command_mode = "plain"

[llm]
base_url = "https://api.openai.com/v1"
model    = "gpt-5.4-mini"

[output]
capture_stdout_bytes = 131072
capture_stderr_bytes = 262144

Quick start

Start simple, stay explicit, and keep the session alive.

go build -o shellia .
./shellia

# one-shot
./shellia "run git status"

# create config
./shellia config init

# direct command
!pwd

# manual shell loop
:shell
pwd
cd project
:ai

# ask Shellia to plan
check if brew has outdated packages

# edit a proposed step before running it
press e at confirmation

Start in prompt mode when you want planning, confirmations, and a readable audit of what Shellia is about to do.

Switch to ! for one direct command, or :shell when you want a manual loop in the same live context. The point is not to hide the terminal. The point is to make it easier to drive.

  • Use prompt mode for multi-step tasks or when output from one command affects the next.
  • Use e if a proposed command is close but needs your final tweak.
  • Use i when a proposed step should run interactively in a real terminal session.
  • Keep commands explicit. Shellia works best when you can inspect every step.
